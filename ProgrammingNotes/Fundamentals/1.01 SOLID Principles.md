The SOLID principles are not just important to mobile iOS development, but to any software development. Whether you move on to a different engineering path or not, being familiar with these concepts, even if you can't remember the exact acronym will be massively beneficial.

1.  Single Responsibility: Every class should have their own responsibility
2. Open-Close: Software entities should be open for extension but closed for modification.
3. Liskov Substitution: Functions that use pointers or references to base classes must be able to use objects from derived classes without knowing it.
4. Interface Segregation: Clients should not be forced to depend on interfaces that they do not use.
5. Dependency Inversion: Depend on abstractions, not concretions.

### Single Responsibility Principle
_A Chef's only job should be to cook, not serve or handle plumbing._

- A type should only have one reason to change- just one focus or purpose.
- When a class handles too many concerns, it becomes harder to maintain, test, or reuse.
- One job per class or struct.

**Negative Example**
- `Order` 's responsibility is to process order data calculations *and* logging.

```
class Order {
    var items: [Double]
    init(items: [Double]) {
        self.items = items
    }
    
    func calculateTotal() -> Double {
        items.reduce(0, +)
    }

    func printReceipt() {
        print("Total: \(calculateTotal())")
    }
}
```

**Positive Example**
- `Order`'s responsibility is to manage order data and calculations.
- `ReceiptPrinter`'s responsibility is to handle logging.

```
class Order {
    var items: [Double]

    init(items: [Double]) {
        self.items = items
    }

    func calculateTotal() -> Double {
        items.reduce(0, +)
    }
}
```

```
class ReceiptPrinter {
    func printReceipt(for order: Order) {
        print("Total: \(order.calculateTotal())")
    }
}
```

in the positive example, each type has *one reason to change*, satisfying the Single Responsibility Principle.

### Open-Close Principle

_You can add an HDMI to a television without having to rewire the TV._

- Your code should be open to adding features, but closed to rewriting existing behavior.
- Use protocols or base classes so you can add new functionality without rewriting old code.

**Negative Example**
- Adding new shapes requires editing the `AreaCalculator`.
- Growing conditional chain wherein code becomes both harder to read and maintain.

```
class AreaCalculator {
    func area(of shape: Any) -> Double {
        if let circle = shape as? Circle {
            return Double.pi * circle.radius * circle.radius
        } else if let square = shape as? Square {
            return square.side * square.side
        }
        return 0
    }
}
```

```
struct Circle {
    let radius: Double
}
	
struct Square {
    let side: Double
}
```

**Positive Example**
- Adding additional shapes, say triangle,  only requires implementation of a new struct `Triangle: Shape`
- Modify calculator no longer needed, each struct makes its own.

```
protocol Shape {
    func area() -> Double
}
```

```
struct Circle: Shape {
    let radius: Double

    func area() -> Double {
        Double.pi * radius * radius
    }
}

struct Square: Shape {
    let side: Double

    func area() -> Double {
        side * side
    }
}
```

```
class AreaCalculator {

    func area(of shape: Shape) -> Double {
        shape.area()
    }
}
```

### Liskov's Substitution Principle
_A vending machine labeled "Chips" shouldn't eject fireworks._

**Negative Example**
- These classes cannot perform the required functionality.
- Substitution breaks because a `Penguin` cannot behave like `Bird`, leading to runtime failures.

```
class Bird {
    func fly() {
        print("Flying")
    }
}

class Penguin: Bird {
    override func fly() {
        fatalError("Penguins cannot fly")
    }
}
```

```
let bird: Bird = Penguin()
bird.fly() // Crash
```

**Positive Example**
- Separate capabilities into appropriate abstractions.
- `Sparrow` can fly, `Penguin` cannot. Therefore, no subclass pretends to support behavior it cannot fulfill.
  
```
protocol Bird { }

protocol FlyingBird {
    func fly()
}
```

```
class Sparrow: Bird, FlyingBird {
    func fly() {
        print("Flying")
    }
}

class Penguin: Bird {
    func swim() {
        print("Swimming")
    }
}
```

### Interface Segregation Principle
**Negative Example**
_Don't give someone a swiss army knife, if all they need is a screwdriver._

- Includes forced functions like `eat()`.
- Includes behavior that is meaningless.
  
```
protocol Worker {
    func work()
    func eat()
}
```

```
class Human: Worker {
    func work() {
        print("Working")
    }

    func eat() {
        print("Eating lunch")
    }
}

class Robot: Worker {
    func work() {
        print("Working")
    }

    func eat() {
        fatalError("Robots do not eat")
    }
}
```

**Positive Example**

- Splits the large protocol into smaller, more easily implemented protocols.
- `Human` has `Workable` and `Eatable`, while `Robot` only has `Eatable`.

```
protocol Workable {
    func work()
}

protocol Eatable {
    func eat()
}
```

```
class Human: Workable, Eatable {
    func work() {
        print("Working")
    }

    func eat() {
        print("Eating lunch")
    }
}

class Robot: Workable {
    func work() {
        print("Working")
    }
}
```

### Dependency Inversion Principle
**Negative Example**
_You plug a device into a wall outlet, you don't care where the electricity is coming from._

- `NotificationService` requires `EmailSender`, therefore any changes to `EmailSender` would be catastrophic for `NotificationService`.
- Cannot easily extend functionality like to something like SMS.
- Difficult to test and mock.

```
class EmailSender {
    func send(message: String) {
        print("Sending email: \(message)")
    }
}

class NotificationService {
    let emailSender = EmailSender()

    func notify() {
        emailSender.send(message: "Hello!")
    }
}
```

**Positive Example**

- Loose coupling, meaning changes won't be catastrophic.
- Add additional message functionality such as `PushSender` without modifying existing code.
- Easily testable classes by implementing `MessageSender`

```
protocol MessageSender {
    func send(message: String)
}
```

```
class EmailSender: MessageSender {
    func send(message: String) {
        print("Sending email: \(message)")
    }
}

class SMSSender: MessageSender {
    func send(message: String) {
        print("Sending SMS: \(message)")
    }
}
```

```
class NotificationService {

    let sender: MessageSender

    init(sender: MessageSender) {
        self.sender = sender
    }

    func notify() {
        sender.send(message: "Hello!")
    }
}
```
